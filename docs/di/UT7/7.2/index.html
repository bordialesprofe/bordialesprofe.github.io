7.2 Estructura básica de una app Flutter

En este apartado repasamos brevemente cómo se organiza una aplicación Flutter y cuál es el papel de los widgets fundamentales que veremos repetidamente a lo largo de la unidad. Este recordatorio permite contextualizar el trabajo que realizaremos a continuación.

1️⃣ El archivo main.dart como punto de entrada​

Toda aplicación Flutter comienza en el archivo situado en: lib/main.dart

Este archivo contiene la función principal del programa:

void main() {
  runApp(const MainApp());
}
Sobre el archivo main.dart y el código inicial

El archivo main.dart contiene el punto de entrada de toda aplicación Flutter.
El código generado por defecto:

  • No debe modificarse, salvo para cambiar el widget raíz.
  • Llama a runApp(), que indica a Flutter qué widget debe dibujar primero en pantalla.
  • Ese primer widget (por ejemplo, MainApp o MyApp) se conoce como widget raíz.
  • El import import 'package:flutter/material.dart'; carga la biblioteca principal de Flutter, que permite utilizar widgets como MaterialApp, Scaffold, Text, Center, etc.

En esta unidad trabajaremos principalmente dentro del widget raíz y en las pantallas que él cargue, pero la estructura inicial del método main() se mantiene siempre igual.

2️⃣ El papel de runApp() y MaterialApp​

La función runApp() le indica a Flutter qué widget debe dibujar primero en pantalla.
Ese primer widget se conoce como widget raíz y normalmente es una clase que extiende StatelessWidget o StatefulWidget.

Stateless vs Stateful
  • StatelessWidget: su contenido no cambia. Ideal para textos, iconos o estructuras fijas.
  • StatefulWidget: su contenido puede cambiar durante la ejecución. Se utiliza para contadores, formularios o elementos interactivos.

El widget raíz suele devolver un MaterialApp, que actúa como contenedor global de la aplicación.
Este widget configura aspectos esenciales:

  • El tema general (colores, tipografía)
  • El título de la aplicación
  • Las rutas de navegación
  • La pantalla inicial mediante la propiedad home

Además, dentro del widget raíz sobrescribimos el método build() utilizando @override, lo que nos permite definir la interfaz que debe mostrarse.
El constructor por defecto (const MiApp({super.key})) marca el widget como inmutable y permite recibir una clave opcional que Flutter utiliza para gestionar el árbol de widgets.

Ejemplo:

class MainApp extends StatelessWidget {
  const MainApp({super.key});

  @override
  Widget build(BuildContext context) {
    return const MaterialApp(
      home: Scaffold(
        body: Center(
          child: Text('¡Hola, Mundo!'),
        ),
      ),
    );
  }
}

Este modelo se utiliza para proyectos pequeños. Más adelante aprenderemos a separar la pantalla inicial en una clase independiente (como HomePage) para lograr una organización más limpia.

3️⃣ Flujo de arranque de una app Flutter​

Cuando se ejecuta una aplicación Flutter, el proceso inicial sigue una serie de pasos bien definidos.
Comprender este flujo permite entender por qué la interfaz se organiza en forma de árbol y cómo se construyen las pantallas.

Los pasos principales del arranque son:

  • 1. main.dart: es el archivo donde comienza la ejecución del programa.
  • 2. main(): función principal que se ejecuta al iniciar la app.
  • 3. runApp(): recibe el widget raíz y se lo entrega al motor de Flutter para empezar a dibujar la interfaz.
  • 4. Widget raíz: primer widget del árbol (normalmente MainApp). Se encarga de devolver un MaterialApp.
  • 5. MaterialApp: configura elementos globales como el tema visual y la pantalla inicial.
  • 6. home: define la primera pantalla que verá el usuario.
  • 7. Scaffold: estructura básica de esa pantalla, sobre la que se construirán los demás widgets.
  • 8. Árbol de widgets: el resto de widgets (Text, Icon, Column, Container…) se organizan jerárquicamente a partir del Scaffold.

Este flujo constituye el punto de partida de cualquier interfaz creada con Flutter y explica cómo se va construyendo visualmente la aplicación desde el nivel más alto hasta cada detalle de la pantalla.

4️⃣ El concepto clave: todo es un widget​

En Flutter, absolutamente todo lo que aparece en pantalla es un widget.
Un widget no es solo un botón o un texto: también lo son la pantalla, los márgenes, los colores, las filas, las columnas y prácticamente cualquier elemento visual o estructural.

Este enfoque se basa en tres ideas fundamentales:

  • 1. La interfaz se construye mediante composición.
    En lugar de modificar elementos, Flutter crea nuevos widgets combinando otros más pequeños.

  • 2. Cada widget devuelve otro widget.
    El método build() de un widget siempre devuelve otro widget, que a su vez contiene más widgets.

  • 3. El resultado final es un árbol jerárquico.
    La interfaz se organiza en forma de árbol, donde cada widget tiene un widget padre y, opcionalmente, varios widgets hijo.

Gracias a esta filosofía, Flutter permite construir interfaces complejas a partir de piezas simples, de manera modular y muy predecible.

Este concepto es esencial para comprender los widgets básicos que veremos a continuación.

5️⃣ La idea del árbol visual​

Dado que en Flutter todo es un widget, la interfaz completa de una aplicación se organiza en forma de árbol visual.
Cada widget ocupa un lugar dentro de esta estructura jerárquica y su apariencia depende del widget padre que lo contiene.

Este modelo se apoya en las siguientes ideas:

  • Relación padre–hijo: un widget puede contener uno o varios widgets hijo. Por ejemplo, Scaffold contiene AppBar, body, FloatingActionButton…

  • Composición progresiva: widgets simples (como Text, Icon o SizedBox) se combinan dentro de widgets de estructura (Row, Column, Container) para formar elementos más complejos.

  • Responsabilidad del widget padre: el widget padre controla la organización, alineación, tamaño y disposición de sus widgets hijo.
    Por ejemplo, una Column organiza sus hijos de arriba abajo; un Row, de izquierda a derecha.

  • Actualización eficiente: cuando cambia un widget hijo, Flutter solo reconstruye esa parte del árbol, sin necesidad de redibujar toda la interfaz.

Comprender esta jerarquía es fundamental para construir interfaces claras y bien organizadas.
En el siguiente apartado comenzaremos a trabajar con los widgets básicos más utilizados en Flutter.

Ejemplo conceptual:

MaterialApp
  └── Scaffold
        └── Column
              ├── Text
              ├── Row
              │     ├── Icon
              │     └── Text
              └── FloatingActionButton
Organización recomendada del código

En los primeros ejemplos podemos colocar el Scaffold directamente dentro de MaterialApp.
Sin embargo, en aplicaciones reales es más habitual separar la pantalla principal en un widget propio, como HomePage.
Esta estructura hace que el código sea más claro, más fácil de mantener y permite añadir más pantallas en el futuro.

Ejemplo simplificado de organización inicial:

void main() {
  runApp(const MainApp());
}

class MainApp extends StatelessWidget {
  const MainApp({super.key});

  @override
  Widget build(BuildContext context) {
    return const MaterialApp(
      home: HomePage(),
    );
  }
}

class HomePage extends StatelessWidget {
  const HomePage({super.key});

  @override
  Widget build(BuildContext context) {
    return const Scaffold(
      body: Center(
        child: Text('Hola Flutter'),
      ),
    );
  }
}
