8.4.4 Widgets clave para adaptar diseños

Además de usar MediaQuery o LayoutBuilder para decidir la estructura general, en Flutter es fundamental que los widgets internos sean flexibles.

Si los elementos tienen tamaños fijos, el diseño puede:

  • desperdiciar espacio en pantallas grandes
  • romperse en pantallas pequeñas (overflow)

Por eso, Flutter incluye widgets que ayudan a repartir espacio y mantener el layout adaptable.

1️⃣ Expanded (repartir el espacio disponible)​

Expanded se usa dentro de un Row o Column para indicar que un widget debe ocupar el espacio libre que quede.

Es decir, en lugar de asignar un ancho fijo, Flutter reparte el espacio automáticamente entre los widgets que estén dentro de Expanded.

Esto es útil cuando se quiere que un diseño funcione bien tanto en pantallas pequeñas como grandes, sin recalcular tamaños a mano.

Ejemplo Salón de héroes
Row(
  children: [
    Expanded(child: _buildListaHeroes()),
    Expanded(child: _buildDetalleHeroe()),
  ],
)
en este ejemplo

En pantallas grandes, la pantalla puede mostrar lista y detalle a la vez.
Expanded permite que ambos paneles se ajusten al ancho disponible.

  • No hay anchos fijos.
  • El reparto del espacio se adapta automáticamente a la pantalla.

2️⃣ Flexible (flexible sin ocupar todo el espacio)​

Flexible también se usa dentro de un Row o Column, y sirve para que un widget pueda adaptarse al espacio disponible, pero con más control que Expanded.

La diferencia principal es:

  • Expanded obliga a ocupar todo el espacio libre.
  • Flexible permite que el widget ocupe solo lo que necesite, sin desbordar, y se adapte si falta espacio.

Esto resulta útil cuando un elemento no debería “estirarse” demasiado, pero sí necesita poder encogerse si la pantalla es pequeña.

Ejemplo Salón de héroes
Row(
  children: [
    Flexible(
      child: Text(
        heroeSeleccionado.nombre,
        overflow: TextOverflow.ellipsis,
      ),
    ),
    const SizedBox(width: 12),
    Expanded(
      child: Text(heroeSeleccionado.descripcion),
    ),
  ],
)
en este ejemplo

En el panel de detalle se muestran:

  • el nombre del héroe (no debería estirarse)
  • la descripción (puede ocupar más espacio)

El nombre se coloca dentro de Flexible para que se adapte sin forzar el layout. Si no cabe, se recorta con ellipsis.

La descripción se coloca dentro de Expanded para ocupar el espacio restante disponible.

3️⃣ Spacer (separar elementos sin medidas fijas)​

Spacer se usa dentro de un Row o Column para crear espacio flexible entre widgets.

Es una alternativa a poner márgenes o SizedBox con valores fijos cuando lo que se busca es que la separación se adapte al tamaño de pantalla.

En otras palabras, Spacer “empuja” widgets hacia los extremos sin necesidad de calcular distancias.

Ejemplo Salón de héroes
Row(
  children: [
    Text(heroe.nombre),
    const Spacer(),
    IconButton(
      icon: const Icon(Icons.star_border),
      onPressed: () {},
    ),
  ],
)
en este ejemplo

El Spacer empuja el botón hacia la derecha, manteniendo el nombre a la izquierda. La separación se ajusta automáticamente al ancho disponible, por lo que funciona mejor que usar una distancia fija en pantallas pequeñas y grandes.

4️⃣ Wrap (cuando no cabe en una fila)​

Wrap es útil cuando se quiere colocar varios elementos en una fila, pero sin riesgo de overflow.

A diferencia de Row, que intenta mantener todos los widgets en una sola línea, Wrap permite que los elementos salten a la siguiente línea si no caben.

Esto es muy práctico en diseño adaptativo para:

  • listas de etiquetas
  • chips de categorías
  • botones pequeños en grupo
Ejemplo Salón de héroes
Wrap(
  spacing: 8,
  runSpacing: 8,
  children: [
    Chip(label: Text(heroe.clase)),
    Chip(label: Text(heroe.rareza)),
    Chip(label: Text(heroe.arma)),
    Chip(label: Text(heroe.region)),
  ],
)
en este ejemplo

Se muestran varios datos del héroe como etiquetas. Si la pantalla es estrecha, Wrap coloca algunos chips en la siguiente línea en lugar de provocar overflow. Si la pantalla es grande, los chips caben en una sola fila.

5️⃣ Evitar overflow: Scroll cuando el contenido no cabe​

Uno de los errores más comunes al adaptar diseños en Flutter es el overflow, cuando el contenido no entra en el espacio disponible.

Esto suele ocurrir cuando:

  • hay mucho contenido en vertical en pantallas pequeñas
  • el teclado aparece y reduce el alto disponible
  • se usan Column con contenido largo sin scroll

Una solución básica es envolver el contenido en un widget con scroll, como SingleChildScrollView.

Ejemplo Salón de héroes
SingleChildScrollView(
  padding: const EdgeInsets.all(16),
  child: Column(
    crossAxisAlignment: CrossAxisAlignment.start,
    children: [
      Text(heroe.nombre),
      const SizedBox(height: 12),
      Text(heroe.descripcion),
      const SizedBox(height: 12),
      Wrap(
        spacing: 8,
        runSpacing: 8,
        children: [
          Chip(label: Text(heroe.clase)),
          Chip(label: Text(heroe.rareza)),
          Chip(label: Text(heroe.arma)),
        ],
      ),
    ],
  ),
)
en este ejemplo

El panel de detalle puede contener mucho texto y varias etiquetas. En pantallas pequeñas, SingleChildScrollView permite desplazarse hacia abajo en lugar de provocar un overflow.

6️⃣ Resumen: ¿Qué widget usar en cada caso?​

En diseño adaptativo no basta con cambiar la estructura general del layout.
También es necesario que los widgets internos se comporten de forma flexible.

En este apartado se han visto las herramientas más importantes:

  • Expanded → Para repartir el espacio disponible dentro de un Row o Column.
  • Flexible → Para permitir que un widget se adapte sin obligarlo a ocupar todo el espacio.
  • Spacer → Para crear separaciones dinámicas sin usar medidas fijas.
  • Wrap → Para evitar overflow horizontal cuando varios elementos no caben en una fila.
  • SingleChildScrollView → Para permitir desplazamiento cuando el contenido vertical no cabe.
Idea clave

El diseño adaptativo no consiste solo en cambiar de layout según el ancho.
Consiste en construir interfaces que puedan ajustarse al espacio disponible sin romperse.

En la práctica, lo habitual es combinar:

  • MediaQuery o LayoutBuilder para decidir la estructura.
  • Widgets flexibles para que el contenido se adapte correctamente dentro de esa estructura.
