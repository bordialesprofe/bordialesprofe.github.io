Fase 1 - Análisis y definición del proyecto

En esta fase identificarás el problema que quieres resolver, definirás a quién va dirigida tu aplicación y establecerás las características mínimas de tu proyecto. Esta fase marca el punto de partida y te permitirá tomar decisiones fundamentadas para avanzar con seguridad en el resto del módulo.

1️⃣ Qué debes hacer en esta fase​

1) Definir la idea del proyecto y el ODS relacionado

  • Selecciona un problema o necesidad real.
  • Elige un ODS relacionado.
  • Explica qué aporta tu proyecto a ese ODS.
  • Resume tu propuesta de forma clara.

2) Analizar al usuario y definir requisitos

  • Identifica a tu usuario principal (perfil, necesidades y objetivos).
  • Describe qué problema tiene y cómo tu proyecto lo ayudará.
  • Redacta requisitos funcionales (qué hará tu aplicación).
  • Redacta requisitos no funcionales (calidad, accesibilidad, rendimiento…).

3) Realizar un análisis externo básico

  • Identifica el sector en el que se enmarca tu proyecto.
  • Busca 3 competidores directos o similares.
  • Expón oportunidades (qué aporta tu proyecto).
  • Añade una nota breve sobre aspectos legales (protección de datos, PRL básica).
  • Menciona posibles ayudas, programas o iniciativas relacionadas.

2️⃣ Entregables de la fase​

📄 Entregable 1 — Idea + ODS + necesidad​

Tipo de archivo: PDF

Nombre recomendado: F1_Actividad1_Idea_Proyecto_TuNombre.pdf

Contenido obligatorio:

  1. Idea del proyecto
  2. Problema o necesidad detectada
  3. ODS seleccionado
  4. Justificación del ODS
  5. Resumen del proyecto
Presentación en clase + Acta de propuestas

Este entregable incluye una presentación breve en clase.
Además, el grupo deberá elaborar un acta con las propuestas de mejora realizadas a cada alumno durante la sesión.
El acta se entregará junto con el PDF y formará parte de la evidencia del hito.

Cómo redactarlo

Puedes crearlo en Google Docs, Word o LibreOffice, pero debes exportarlo en PDF antes de entregarlo.

Extensión recomendada

1–2 páginas son suficientes. Sé claro, concreto y evita tecnicismos innecesarios.

Evita este error

No describas el ODS de forma genérica. Explica qué relación real tiene con tu proyecto.

📄 Entregable 2 — Perfil de usuario y requisitos​

Tipo de archivo: PDF

Nombre recomendado: F1_Actividad2_Perfil_Requisitos_TuNombre.pdf

Contenido obligatorio:

  1. Perfil de usuario principal

    • Datos básicos (edad aproximada, ocupación, nivel digital).
    • Necesidades, objetivos y motivaciones.
    • Puedes incluir una imagen o icono representativo.

  2. Análisis de la realidad del usuario

    • Qué problema tiene.
    • En qué contexto utilizará tu aplicación.

  3. Requisitos funcionales (RF)

    • Lista numerada.
    • Describe qué debe permitir hacer la aplicación.

  4. Requisitos no funcionales (RNF)

    • Accesibilidad, rendimiento, seguridad, usabilidad…
    • Indica solo lo que realmente necesitarás.

  5. Conclusión

    • Qué implicaciones tendrá este análisis en tu diseño.
Presentación en clase + Acta de propuestas

Este entregable incluye una presentación breve en clase.
Además, el grupo deberá elaborar un acta con las propuestas de mejora realizadas a cada alumno durante la sesión.
El acta se entregará junto con el PDF y formará parte de la evidencia del hito.

Consejo para definir requisitos

Los requisitos deben ser claros, medibles y concretos.
Ejemplo correcto: “El sistema permitirá crear, editar y eliminar tareas.”
Ejemplo incorrecto: “El sistema será intuitivo y funcionará bien.”

Herramientas recomendadas
  • Canva para crear el perfil de usuario con imagen.
  • Notion / Trello para organizar requisitos.
  • Google Docs / Word para redactar el documento final.
Evita este error

No escribas requisitos como frases vagas (“la app será bonita”).
Todo requisito debe poder comprobarse durante el desarrollo o en las pruebas.

📄 Entregable 3 — Análisis externo del proyecto​

Tipo de archivo: PDF

Nombre recomendado: F1_Actividad3_Analisis_Externo_TuNombre.pdf

Contenido obligatorio:

  1. Sector y empresas tipo

    • Define en qué sector se ubica tu aplicación.
    • Incluye 2–3 ejemplos de empresas o servicios similares.

  2. Competencia directa e indirecta

    • Selecciona 2–3 aplicaciones que resuelvan problemas parecidos.
    • Compara brevemente qué hacen y qué no hacen.

  3. Oportunidades

    • Mini DAFO (solo fortalezas y oportunidades)
      o
    • Lista de 3–4 ventajas de tu propuesta frente al mercado.

  4. Aspectos legales esenciales

    • Protección de datos (obligatorio siempre).
    • PRL mínima si aplica (normalmente riesgo muy bajo).
    • Obligaciones fiscales solo si el proyecto las implica (indicarlas brevemente).

  5. Ayudas o programas públicos relacionados

    • Menciona 1–2 iniciativas, convocatorias o programas (aunque sea de forma general).
Cómo facilitar este análisis

Busca solo lo imprescindible. No es un estudio académico.
Con 1–2 párrafos por sección es suficiente.

Herramientas recomendadas
  • Google Docs para elaborar tablas o comparativas simples.
  • Google Sheets si prefieres hacer comparaciones en tabla (insértala luego en el PDF).
Evita este error

No copies descripciones de empresas o leyes desde Internet.
Debes adaptarlas al contexto de tu proyecto de forma breve y clara.

3️⃣ Qué partes se incorporan a la memoria final​

El trabajo realizado en esta fase se incorporará principalmente al apartado 3. Análisis del proyecto de la memoria final.

En este apartado se describe el análisis previo realizado antes de diseñar y desarrollar la aplicación. El trabajo desarrollado durante esta fase servirá para justificar el problema detectado, definir el perfil de usuario y establecer los requisitos que deberá cumplir el sistema.

Además, permitirá contextualizar el proyecto dentro de su sector y analizar aspectos relevantes como la competencia, las oportunidades existentes o los requisitos legales que pueden afectar a la aplicación.

Para conocer con más detalle qué debe incluir este capítulo, consulta el apartado Estructura de la memoria final.

4️⃣ Sugerencias para completar la fase​

Empieza por un problema pequeño

Elige una idea realista para el tiempo y recursos disponibles. Una aplicación demasiado grande será difícil de terminar.

Cómo redactar requisitos

Los requisitos deben ser breves, claros y verificables.
Ejemplo correcto: “El sistema permitirá editar el perfil del usuario.”

Evita añadir funcionalidades extra

Muchas ideas fallan porque se intenta abarcar demasiado.
Primero completa el mínimo del proyecto; después, si puedes, amplías.

Revisar tu idea con otros compañeros ayuda

Comparar ideas y requisitos con otros estudiantes te permite detectar problemas antes de avanzar a la fase de diseño.
